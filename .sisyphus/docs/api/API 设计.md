# API 设计（已实现）

## 目标

API 需要支撑前端完成四类操作：

1. 查看抓包列表
2. 查看某个包的详情
3. 筛选异常包
4. 查看某个包与前一个包的比较内容

API 的设计重点不是通用性，而是围绕本工具的分析流程保持清晰和稳定。

---

## API 分组

建议按功能分成三组：

- 抓包写入
- 列表 / 详情读取
- 比较 / diff 读取

---

## 1. 抓包写入接口

### `POST /api/packets`（已实现）

用途：写入一个新抓到的请求包。

这个接口对应后端抓包流程把新包落入系统。

请求体建议至少包含：

- 原始请求信息
- 原始响应信息
- 时间信息
- 能提取到的缓存统计

返回值建议至少包含：

- 新包 ID
- 顺序号
- 是否为异常包
- 是否可比较

这样后续前端在实时更新列表时，可以直接使用返回摘要。

---

## 2. 列表接口

### `GET /api/packets`（已实现）

用途：获取请求包列表。

当前支持：

- 默认顺序返回
- 只看异常包的筛选

当前尚未实现分页或分段加载。

每个列表项建议返回：

- `id`
- `sequence`
- `captured_at`
- `host`
- `path`
- `cached_tokens`
- `total_tokens`
- `cache_ratio`
- `is_cache_drop`
- `compare_block_reason`

列表接口返回的是“概要”，不必把所有原始 body 都带回。

---

## 3. 详情接口

### `GET /api/packets/{id}`（已实现）

用途：获取某个请求包的完整详情。

详情接口应返回：

- 基本信息
- Burp/Yakit 风格 raw 请求 / 响应文本
- 原始请求 / 响应内容
- 格式化后的可读文本
- 缓存统计
- 比较状态

详情接口的目标是让前端在选中某个包后，有足够数据展示右侧详情面板。

当前详情接口包含 `raw_request_text` 与 `raw_response_text`。这两个字段是前端 raw 视图的直接数据源，表示抓包入口按 HTTP 报文格式保存的请求与响应文本；前端展示时不再对它们做 JSON 格式化、摘要抽取或 diff 加工。

---

## 4. diff / 比较接口

### `GET /api/packets/{id}/diff-prev`（已实现）

用途：获取某个请求包与前一个包之间的比较输入。

返回结果建议明确区分三种情况：

### 情况 A：可比较

返回：

- 当前包 ID
- 前一个包 ID
- 当前包可读文本
- 前一个包可读文本
- 比较类型（例如 JSON / 文本）

这里返回的核心不是“已经渲染好的 diff HTML”，而是供现有 diff 页面消费的输入内容。

### 情况 B：无前一个包

返回明确状态，告诉前端该包没有前序对象。

### 情况 C：被规则剔除

例如与前一个包间隔超过 4 分钟时，应返回明确状态和原因。

这样前端才能在界面里清楚告诉用户：不是系统坏了，而是这个包不参与当前直接比较。

## 额外已实现接口

### `DELETE /api/packets`

用途：清空当前所有抓包历史。

当前前端“清空历史”按钮直接使用这个接口。

用途：给页面顶部状态区提供摘要。

建议返回：

- 总请求数
- 异常包数
- 可比较包数
- 当前是否正在抓包
- 最近一次抓包时间

这个接口的作用是避免前端自己从大量列表数据里反复汇总。

---

## 返回风格要求

API 返回应优先强调：

- 状态明确
- 字段稳定
- 便于前端直接消费

特别是比较接口，不能只返回空值让前端自己猜原因；应明确告诉前端：

- 可比较
- 不可比较
- 为什么不可比较

---

## 与 diff 页面衔接的要求

由于 diff 展示必须沿用 `/root/_/misc/diff/diff.html`，所以 API 应优先提供“比较输入数据”，而不是自己生成另一套 diff 结果。

也就是说，API 的职责更接近：

- 提供当前包文本
- 提供前一个包文本
- 提供文本类型和状态

至于具体 diff 呈现，仍由既有 diff 页面方式处理。

---

## 当前 API 边界

当前已固定为 REST 风格 API，未实现：

- 鉴权
- 推送更新
- 分页
- 批量删除以外的运维接口
