# 数据模型设计（已实现）

## 目标

数据库需要支撑三件事：

1. 持续记录每一个请求包
2. 支持按顺序浏览与筛选
3. 支持当前包与前一个包之间的比较视图

数据库不是只为了“存日志”，而是为了支撑后续查询、筛选、标记和 diff 展示。

---

## 核心实体

建议至少有一个核心实体：`request_packets`。

这个实体表示“抓到的一个请求包及其对应响应”。

### request_packets（已实现）

建议包含以下字段：

- `id`
- `sequence`
- `captured_at`
- `host`
- `path`
- `method`
- `raw_request_text`
- `raw_response_text`
- `request_headers_text`
- `request_body_text`
- `response_headers_text`
- `response_body_text`
- `formatted_request_body_text`
- `formatted_response_body_text`
- `cached_tokens`
- `total_tokens`
- `cache_ratio`
- `is_cache_drop`
- `comparable_prev_packet_id`
- `compare_block_reason`

这里的一个关键点是：

- 原始内容需要保留
- 归一化后的统计字段也要保留

否则后续要么无法回溯原始包，要么每次查询都要重复解析响应内容。

---

## 关键字段说明

### `sequence`

用于表达顺序关系。

因为当前产品功能是按顺序看包、拿当前包与前一个包比较，所以顺序是数据库层面的核心字段，而不是纯 UI 临时逻辑。

### `cached_tokens` / `total_tokens`

这两个字段用于支撑缓存命中分析。

它们应作为可查询字段存在，而不是每次只从原始响应 JSON 临时解析。

当前实现中，这两个字段在抓包入口阶段尝试从以下来源提取：

- JSON 响应体里的 `usage`
- SSE/流式 `data:` 块中的 `usage`
- OpenAI Responses API 的 `usage` / `response.usage`
- Gemini 的 `usageMetadata`
- 响应头中的 token 统计字段

### `cache_ratio`

这是派生字段，用于提高列表和筛选性能。

### `is_cache_drop`

这是派生布尔字段，表示该请求是否满足：

`cached_tokens < total_tokens * 0.5`

把它存下来可以简化筛选与前端状态展示。

### `comparable_prev_packet_id`

用于记录“当前包应当和谁比较”。

当前功能里这个对象通常是前一个包，但数据库层显式存出来会更利于稳定查询和后续扩展。

### `compare_block_reason`

用于说明某个包为什么不参与比较。

至少应支持一种原因：

- 与前一个包间隔超过 4 分钟

有了这个字段，前端就能明确区分：

- 没有比较对象
- 被时间规则剔除
- 正常可比较

## 原始内容与展示内容分层

数据库里建议区分两类内容：

### 1. 原始内容

例如：

- Burp/Yakit 风格完整请求报文文本
- Burp/Yakit 风格完整响应报文文本
- 原始请求体
- 原始响应体
- 原始请求头
- 原始响应头

这部分用于保真和后续回放、排查。

### 2. 归一化 / 展示内容

例如：

- 格式化后的 JSON 文本
- 缓存统计字段
- 比较状态字段

这部分用于提升前端显示效率和用户可读性。

这样可以避免前端每次点击都重新做重格式化或统计提取。

---

## 关于 diff 数据的存储策略

当前更适合存“比较输入”，而不是直接把 diff 结果固化进数据库。

也就是说，数据库优先保证：

- 当前包的可读文本
- 前一个比较对象的可读文本
- 它们的关系是否成立

而不是提前把某个 diff 结果永久存死。

原因是 diff 展示方式已有固定页面约束，后续很可能仍需在展示层动态生成。

---

## 查询目标

数据库需要支持的典型查询包括：

1. 按顺序查询请求列表
2. 查询异常包列表
3. 查询某个包的详情
4. 查询某个包的前一个可比较包
5. 查询某个包为什么不可比较

这意味着数据模型要为“列表效率”和“详情准确性”同时服务。

---

## 当前数据库边界

当前已固定为：

- SQLite 单文件数据库：`data/packets.sqlite3`
- 无 ORM
- 直接用 `sqlite3` 建表与查询

仍未实现：

- 数据迁移版本管理
- 会话维度隔离
- 多表拆分
