# OpenCode 请求抓包监控工具（已实现）

## 功能目标

这个工具用于持续观察 OpenCode 发出的真实请求，并帮助用户快速发现缓存命中明显变差的请求。

这个工具当前要解决的是“看到请求”和“比较请求”，而不是解释请求变化的来源。

---

## 功能 1：持续抓取请求

工具应能够持续抓取 OpenCode 发出的请求，并把抓到的请求记录下来，供后续查看。

这里的重点是：

- 能持续抓取
- 能保留抓到的请求
- 用户后续可以看到这些请求

当前实现中，这个功能通过 `mitmdump + mitm_capture.py + FastAPI` 打通：抓到的请求响应会自动写入本地 SQLite。

---

## 功能 2：按顺序展示每个请求包

前端应以有先后顺序的方式展示抓到的请求包。

用户在界面中应能区分：

- 第 1 个包
- 第 2 个包
- 第 3 个包
- ……

也就是说，系统需要把请求包当作一个有顺序的序列来展示，而不是无序散点。

当前实现中，顺序由数据库中的 `sequence` 字段维护，前端列表按倒序展示最新包。

---

## 功能 3：显示缓存相关统计

每个请求包在列表中应显示缓存相关统计信息，以便用户判断当前请求的缓存命中情况。

至少要支持体现两个概念：

- `cached_tokens`
- `total_tokens`

用户需要能从界面上直接看出某个包的缓存命中情况，而不必手动翻原始响应。

当前实现中，页面列表与详情区都会展示 `cached_tokens`、`total_tokens` 与命中比例。系统会尝试从普通 JSON 响应、SSE/流式 `data:` 块、OpenAI Responses API、Gemini `usageMetadata` 和响应头中提取统计，但不同供应商的真实格式仍可能继续影响提取效果。

---

## 功能 4：标记缓存命中显著变差的包

如果某个后续请求包满足下面条件：

`cached_tokens < total_tokens * 0.5`

则这个包应被视为“缓存命中显著变差包”。

对这种包，界面中应执行：

- **标红显示**

这样用户在列表中可以快速注意到这些异常包。

---

## 功能 5：支持筛选异常包

前端应允许用户筛选出“缓存命中显著变差包”。

也就是说，用户应能在界面中快速只看这些异常请求，而不是必须手动从完整列表里逐个寻找。

当前实现中，顶部按钮可在“显示全部”和“仅看异常包”之间切换。

---

## 功能 6：点击请求查看与前一个请求的 diff

当用户点击某个请求包时，应展示这个请求包与**前一个请求包**之间的 diff。

这里的对比关系是：

- 当前包 vs 前一个包

这个 diff 视图的目标是帮助用户直接看到两次请求之间到底差了什么。

如果当前包与前一个包之间的时间间隔**超过 4 分钟**，则这两个包不应进入这类直接对比流程。

原因是时间间隔过长时，缓存未命中可能只是因为缓存窗口没有打中，而不是请求内容本身发生了值得关注的变化。

因此，这类间隔超过 4 分钟的请求对，应被系统**自动剔除**，避免误导用户把时间因素误判为内容差异导致的缓存掉点。

当前实现中，点击某个包会打开全屏抽屉式详情页，并自动加载它与前一个可比较包的 diff；被时间规则剔除时，会显示不可比较原因。

---

## 功能 7：比较前先把 JSON 格式化为人类易读形式

在进行 diff 展示之前，如果比较对象是 JSON 内容，应先把 JSON 格式化为**人类易读**的形式，再进行比较。

目标是让用户在 diff 中看到的不是一大段难以阅读的压缩 JSON，而是结构清晰、缩进明确、便于人工审查的内容。

也就是说：

- 先格式化
- 再比较

---

## 功能 8：diff 展示方式沿用现有 diff 页面

diff 的展示方式必须沿用：

- `/root/_/misc/diff/diff.html`

这里的要求不是“做一个差不多的 diff”，而是：

> **按它现有的 diff 方式来展示。**

用户需要看到的是与现有 diff 页面一致的 diff 行为，而不是另一套新写的 diff 交互。

当前实现中，没有再整页 iframe 复用 `diff.html`，而是直接抽取了其第二页 Monaco diff 初始化与修复逻辑，内嵌到当前前端页面中。

---

## 功能 9：清空历史记录

前端应允许用户一键清空当前抓包历史，并在清空后立即看到空状态界面，而不是继续显示旧的选中项或旧 diff。

当前实现中：

- 顶部有“清空历史”按钮
- 后端提供 `DELETE /api/packets`
- 清空后会重置当前选中项、关闭详情抽屉并刷新统计与列表

---

## 当前功能边界

当前实现已经固定为单实例本地工具，仍保留以下边界：

- 默认按单条顺序流理解请求，不做多会话隔离
- 只抓模型请求路径，不保留无关 MCP 噪音流量
- `cached_tokens` / `total_tokens` 的提取仍依赖上游响应格式，若供应商返回格式完全不带 usage/统计头，则可能为空
