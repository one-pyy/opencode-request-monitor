# FastAPI + SQLite 实现方案（已实现）

## 已确认技术方向

当前项目实现方向确认如下：

- 后端服务：**FastAPI**
- 本地数据库：**SQLite**

这意味着后续实现默认围绕“一个本地 HTTP 服务 + 一个本地 SQLite 数据库”展开，不再继续做后端框架和数据库选型比较。

---

## 为什么这套组合适合当前项目

### FastAPI

适合这个项目的原因：

- 接口层清晰，适合快速搭建抓包查询服务
- 对本地工具型项目足够轻量
- 很适合承载列表、详情、统计、diff 输入这类接口
- 后续如果需要加实时更新能力，也有扩展空间

### SQLite

适合这个项目的原因：

- 本地工具场景下部署成本最低
- 单用户、本地持续记录请求包的场景足够胜任
- 便于直接存请求序列、缓存统计和比较关系
- 调试和导出都简单，方便你手工检查数据

当前项目目标是先把工具做对、做顺，而不是先搭一个复杂分布式后端，所以 SQLite 是合适的第一版落点。

---

## 当前已实现的最小闭环

当前已经实现：

1. 能接收并记录抓到的请求包
2. 能从数据库里读取顺序列表
3. 能识别并标记缓存异常包
4. 能给前端提供某个包与前一个包的比较输入
5. 能让前端按既定 diff 方式展示差异
6. 能清空历史记录
7. 能通过启动脚本一键拉起代理与 Web/API 服务

这里不要求第一版同时解决所有扩展问题，例如多用户、多会话、多项目隔离。

## 当前服务分层

建议服务先按下面几层组织。

### 1. 抓包接收层（已实现）

职责：

- 接收抓到的请求与响应数据
- 做基础清洗
- 抽取缓存统计
- 交给存储层落库

这一层的重点是“把包稳定写进去”，不是做复杂分析。

### 2. 存储层（已实现）

职责：

- 把请求包写入 SQLite
- 维护顺序号
- 维护异常标记
- 维护可比较关系与阻断原因

这一层负责把原始包变成后续查询可用的数据记录。

### 3. 查询层（已实现）

职责：

- 提供列表查询
- 提供详情查询
- 提供异常筛选
- 提供顶部摘要统计

这一层面向前端读取，不应混杂抓包写入逻辑。

### 4. 比较输入层（已实现）

职责：

- 给定某个包 ID
- 找到它的前一个可比较包
- 准备当前包与前一个包的可读文本
- 对 JSON 内容先格式化再返回

这一层不负责重新发明 diff，只负责给既有 diff 页面准备输入。

## FastAPI 服务结构（已实现）

从职责上看，FastAPI 至少应包含这些接口组：

- 抓包写入接口
- 请求列表接口
- 请求详情接口
- 比较输入接口
- 统计摘要接口

这几个接口组已经足够覆盖当前 docs 中定义的功能。

建议原则：

- 写入接口与读取接口分开
- 列表接口只返回摘要
- 详情接口返回完整内容
- diff 接口返回比较输入，不返回另一套自制 diff 结果

## SQLite 数据落点（已实现）

SQLite 在这个项目里不只是日志仓库，而是当前工具的状态中心。

建议数据库至少承担：

- 请求顺序记录
- 缓存统计持久化
- 异常包查询
- 可比较关系记录
- 不可比较原因记录

这意味着数据库要能支撑前端直接查：

- 哪些包是异常包
- 哪些包可以比较
- 哪些包因为超过 4 分钟而被剔除

---

## 关键业务规则在实现中的落点

### 1. 缓存异常判定

规则：

`cached_tokens < total_tokens * 0.5`

建议在写入或入库后尽早派生出 `is_cache_drop`，这样后续列表和筛选都更简单。

### 2. 4 分钟剔除规则

规则：

- 当前包与前一个包间隔超过 4 分钟
- 则这两个包不进入直接 diff 比较流程

建议这条规则也在后端侧形成明确状态，而不是只在前端临时判断。

这样前端更容易稳定显示：

- 可比较
- 不可比较
- 不可比较原因

### 3. JSON 可读化

规则：

- diff 前先把 JSON 格式化为人类易读形式

建议这一步在“比较输入层”完成，而不是每个前端页面各自重复做。

这样可以确保：

- 输出一致
- 不同页面不出现格式化差异
- diff.html 始终拿到已经整理好的输入

## 与前端的衔接方式（已实现）

FastAPI 提供数据，前端负责：

- 展示请求列表
- 做筛选交互
- 展示详情面板
- 把比较输入送进内嵌 Monaco diff 区（逻辑提取自 `diff.html` 第二页）

职责边界应当是：

- 后端负责状态正确
- 前端负责展示清晰

不要把复杂比较判断丢给前端，也不要让后端去生成一整套定制 diff UI。

## 当前运行约定（已实现）

### 端口

- `30499`：mitmproxy / mitmdump 监听端口
- `30500`：FastAPI + Web 前端端口

### 启动入口

- `start.sh`
- `mitm_opencode`（安装在 `~/.local/bin`）

### 代理链

- OpenCode → `30499`
- `30499` 上游默认转发到 `30084`
- 抓包结果自动写入 `30500/api/packets`

### 路径过滤

当前仅保留以下请求路径：

- `/v1/chat/completions`
- `/v1/messages`
- `/v1/responses`
- `/v1beta/models/`

这样可以过滤 `mcp.exa.ai` 等非模型请求噪音。

### 响应格式适配

当前抓包入口与详情展示已覆盖：

- OpenAI Chat Completions 的 JSON / SSE 响应
- OpenAI Responses API 的 JSON / SSE 响应
- Gemini `candidates[].content.parts[].text` 返回文本
- Gemini `usageMetadata` 缓存统计
