# 界面与交互设计（已实现）

## 目标

前端需要比“工具页”更现代化，做到：

- 信息密度高，但不显乱
- 异常请求能一眼看见
- 列表、筛选、详情、diff 之间切换流畅
- 面向人工排查缓存问题，而不是面向通用抓包分析

这里的“现代化”不等于炫技，而是强调：清晰层级、快速定位、少阻碍操作。

---

## 页面结构

当前前端已经落成三块主结构。

### 1. 顶部状态区

用于展示当前抓包与筛选状态，例如：

- 当前是否正在监听
- 当前监听目标或来源摘要
- 已记录请求总数
- 异常包数量
- 当前筛选状态

顶部状态区应该足够简洁，优先回答“现在系统在干什么、抓到了多少东西”。

### 2. 左侧请求列表区（已实现）

这是主导航区域，用于按顺序展示抓到的请求包。

每个列表项至少应体现：

- 顺序号
- 时间
- 目标摘要（如 host / path）
- 缓存统计摘要
- 是否为异常包

异常包要有明显视觉差异，不能仅靠很弱的颜色变化。

### 3. 全屏抽屉式详情 / diff 区（已实现）

当用户选中某个包时，右侧显示：

- 该包基本信息
- 缓存统计
- 与前一个包的 diff 入口或 diff 结果
- Burp/Yakit 风格 raw 报文开关

当前实现不是固定右栏，而是点击列表项后打开全屏抽屉，详情与 diff 一起放在抽屉中。这种方式已经替代了早期的简陋内联面板。

详情区当前提供 `Raw` 开关。开启后，raw 视图凌驾于“发送内容 / 返回内容 / 发送 header”三个普通视图之上，直接展示抓包时保存的 HTTP 请求与响应文本，不再做 JSON5 格式化、最终消息摘要抽取或 diff 加工。

普通“返回内容”视图会尝试从 OpenAI Chat Completions、OpenAI Responses API、Anthropic 和 Gemini 响应中提取最终文本；Gemini 文本来源为 `candidates[].content.parts[].text`。

## 核心交互

### 请求列表浏览

用户进入页面后，默认应先看到按顺序排列的请求列表。

重点要求：

- 默认排序清晰
- 新请求进入后不会让用户失去当前位置
- 用户可以快速定位最新请求和异常请求

### 异常筛选（已实现）

用户应能快速切换：

- 显示全部请求
- 仅显示缓存异常请求

筛选操作应当是轻量级切换，而不是复杂表单操作。

### 点击查看差异（已实现）

点击某个包后，系统应进入该包的分析视角。

如果该包满足可比较条件，则应展示它与前一个包之间的 diff。

如果不满足可比较条件，例如没有前一个包，或因时间间隔过长被剔除，也应明确提示原因，而不是空白。

## 视觉状态要求

### 正常包

正常包应保持低干扰显示，不抢异常包注意力。

### 异常包

异常包应明确标红，但要避免纯错误告警式大面积刺眼红色。更好的做法是：

- 行级强调
- 状态标签
- 数值颜色强化

这样既醒目，也便于长时间审查。

### 不可比较包（已实现）

例如与前一包间隔超过 4 分钟的包，应有独立状态展示，明确告诉用户：

- 它被记录了
- 但不参与当前直接比较

不能把“不可比较”混成“正常”或“异常”。

---

## diff 区要求（已实现）

diff 不是普通文本比对，而是这个工具的核心审查界面之一。

要求：

- 进入方式自然，不绕
- 当比较对象是 JSON 时，先格式化再比较
- diff 区与列表选中状态要保持同步
- diff 的视觉风格要与主体页面协调，但核心 diff 实现方式仍沿用既有 `diff.html` 第二页逻辑

当前实现已经做到：外层页面现代化，diff 内核不另起一套，而是抽取 `diff.html` 第二页的 Monaco diff 初始化与修复逻辑。

开启 raw 视图时不进入 diff 内核，而是用只读 Monaco 文本视图展示裸报文。

## 页面节奏（半实现）

这个工具是“持续产生新数据”的工具，所以前端不能只按静态页面来设计。

需要考虑：

- 新请求不断出现时，列表如何更新
- 用户正在看旧请求时，是否强制跳转
- 异常包出现时，如何提醒但不打断

当前已做到：

- 默认通过手动刷新更新
- 用户选中某个包后会保持当前焦点，除非主动关闭抽屉
- 异常包会在列表中用红色状态强调

尚未实现自动增量刷新或温和提示机制。

---

## 当前前端边界

当前实现已经固定为：

- 单 HTML 文件前端
- Tailwind CDN + 原生 JavaScript
- Monaco Diff Editor 内嵌

未做组件化拆分，也未做更复杂的状态管理。
